## 決定事項

- Context7 MCP は `npx @upstash/context7-mcp@latest` で起動する（他の AWS 系 MCP が `uvx` を使うのとは異なる）
- ドキュメント（plans/specs）は `.claude/docs/` 以下に統一する（リポジトリルートの `./docs/` は使わない）
- Context7 はサードパーティ MCP（Upstash 製）であることをユーザーに確認し、継続採用を選択した

## 捨てた選択肢と理由

- `./docs/plans/` にドキュメント整理: ユーザーが `.claude/docs/` への変更を指示したため破棄 — `.claude/docs/plans/`, `.claude/docs/specs/` を採用
- Context7 を見送る案: ユーザーが「サードパーティか」と確認後、継続採用を選択したため採用

## ハマりどころ

- `git mv docs/plans/.gitkeep .claude/docs/plans/.gitkeep` 実行時に `fatal: pathspec 'docs/' did not match any files` エラー → `git mv` を先に実行していたため `.gitkeep` が既にステージング済みでワーキングツリーに存在しなかった → すでに rename がステージされていたことを `git status` で確認して解消

## 学び

- `.claude/docs/plans/` への `git mv` 後に `git status` を確認すると、既存の計画ファイル（以前のセッションで `.claude/docs/plans/` に直接置かれていたもの）が untracked として現れた。`.claude/docs/plans/` は以前からこのリポジトリに存在していた
- Context7 MCP のツール名は `resolve-library-id` と `query-docs`。エージェント定義では `mcp_context7` と記述する
