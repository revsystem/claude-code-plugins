## 決定事項

- プラグインの MCP 設定を変更した際は、必ず `marketplace.json` のバージョン番号を上げる（キャッシュ無効化のため）

## 捨てた選択肢と理由

- `aws-mcp`（`aws-core@agent-toolkit-for-aws` 提供）を既存の `awslabs.aws-documentation-mcp-server` の重複と見なして削除する案: `aws-mcp` は `https://aws-mcp.us-east-1.api.aws/mcp` ゲートウェイであり、用途が異なる — 削除せずプラグイン版として維持
- PR #20 のみでの修正完結: バージョン番号を上げなかったため `/reload-plugins` でキャッシュが更新されず起動失敗が継続した — PR #22 でバージョン番号更新とリネームを追加して解決

## ハマりどころ

- PR #20 マージ後も `/reload-plugins` で接続失敗が続いた → Claude Code はバージョン番号をキャッシュキーとして使用しており、バージョンが `0.0.4` のままだと旧キャッシュを読み続ける → PR #22 でバージョンを `0.0.5` に上げて解決
- `plugin:aws-cost-analyst:aws-cost-explorer-mcp-server` の接続失敗を認証エラーと誤認しがちだったが、実際は PyPI の yanked によるパッケージ解決失敗 (`uvx` はyankedパッケージのインストールを拒否する)

## 学び

- Claude Code プラグインキャッシュは `~/.claude/plugins/cache/<marketplace>/<plugin>/<version>/` 構造で管理される。バージョン番号を変えない限り、コンテンツ変更後に `/reload-plugins` しても旧キャッシュを使い続ける
- PyPI の yanked パッケージは `uvx` が解決を拒否する。エラーメッセージに `(reason: Superceeded by ...)` が含まれるため後継パッケージが特定できる
- `aws-core@agent-toolkit-for-aws` プラグインが提供する `aws-mcp` は AWS MCP Gateway（SigV4 認証経由で複数AWSサービスにアクセス）であり、`aws-documentation-mcp-server` とは別物
