## 決定事項

- MCP を使う read-only プラグインエージェントは `tools` アロウリストではなく `disallowedTools: Write, Edit` を使う。理由: 公式 plugins-reference のエージェント例がこの方式で、プラグイン MCP のスコープ命名問題（`mcp__plugin_<plugin>_<server>__<tool>`）を回避でき確実に MCP を継承できるため
- skills（ユーザーレベル agent）は allowlist 方式 + `mcp__<server>` を使う。理由: サーバー名が `mcpServers` インライン定義のキーと一致し、スコープ接頭辞が不要でクリーンなため（ユーザー判断）
- terraform-code-reviewer の description は plugin/skills 両方で統合版に統一。MCP 供給方式のみ環境差で最適化する
- article-reviewer は read-only（提案のみ、適用は呼び出し元）。MCP 不要なので allowlist `Read, Grep, Glob`
- バージョンはパッチバンプ: document-reviewer 0.0.4→0.0.5、terraform-code-reviewer 0.1.0→0.1.1、aws-cost-analyst 0.0.5→0.0.6、tech-docs-searcher 0.0.6→0.0.7、metadata 0.0.5→0.0.6

## 捨てた選択肢と理由

- plugin で tools アロウリスト + `mcp__<server>` ベア名: プラグイン配下では実ツール名が `mcp__plugin_<plugin>_<server>__<tool>` とスコープされ、ベア名がマッチするか未検証。スコープ付き名 `mcp__plugin_terraform-code-reviewer_...` はプラグイン名に結合し脆い — plugin は `disallowedTools` を採用
- skills を `disallowedTools` 方式にする案（Claude が提案）: ユーザーが「allowlist のみ修正」を選択。skills はサーバー名がキーと一致しベア名で確実にマッチするため allowlist が最もクリーン — ユーザー判断で allowlist を採用
- article-reviewer のインプレース編集維持: グローバル CLAUDE.md「レビュー役はファイルを修正しない」と不整合 — read-only を採用（ユーザーが AskUserQuestion で選択）

## 学び

- MCP ツールの正しい表記はダブルアンダースコア（`mcp__server` で全ツール、`mcp__server__tool` で個別、`mcp__server__*` ワイルドカード）。`mcp_server`（1本）は不正で、tools アロウリストでは MCP にマッチせずアクセス不能になる（permissions / tools-reference ドキュメントで確定）
- サブエージェントの tools をアロウリスト指定すると、MCP を明示列挙しない限り MCP ツールは使えない。`disallowedTools` デナイリストなら MCP・読み取り系を自動継承し Write/Edit のみブロックできる（公式 sub-agents ドキュメント明記）
- プラグイン提供 MCP の実ツール名は `mcp__plugin_<プラグイン名>_<サーバー名>__<tool>` とスコープされる（セッションの deferred tool 一覧で確認）
- プラグインエージェントは `hooks`/`mcpServers`/`permissionMode` フロントマターを使えない（セキュリティ上無視）。MCP は marketplace.json のプラグインレベル mcpServers で定義する。一方ユーザーレベル agent は `mcpServers` インライン定義が使える
- 公式 plugins-reference のエージェント例自体が `disallowedTools: Write, Edit` を採用しており、プラグイン read-only エージェントの推奨パターンと判断できる
